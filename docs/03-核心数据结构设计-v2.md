# LLMHub 核心数据结构设计（v0.2）

- 数据库：PostgreSQL 16 + TimescaleDB
- 上一版：[`03-核心数据结构设计.md`](03-核心数据结构设计.md)（v0.1，已废弃）

## 数据库分域

| Schema | 用途 |
|---|---|
| `iam` | 用户、API Key、会话 |
| `wallet` | 余额、交易流水、充值、发票 |
| `catalog` | 平台 SKU、定价 |
| `pool` | 厂商主账号、凭据、服务绑定 |
| `metering` | 调用流水（TimescaleDB）、预扣费、聚合视图 |
| `audit` | 管理员操作流水 |

## 资源分层

```
平台侧（用户看到）            上游侧（运营管理）
─────────────────             ───────────────────────────
PlatformCategory  大类         Vendor          厂商
 └ PlatformService  SKU          └ VendorAccount  主账号
                                    └ VendorProduct  业务板块
                                       └ Credential     凭据
                                          └ ServiceBinding  服务绑定

PlatformService.{vendor_product_id, capability, model}  ── 把两条链关联起来
```

代码常量（不进 DB）：`PlatformCategory` · `Vendor` · `VendorProduct` · `Capability`，定义在 Go 包 `app/internal/catalog/`。
DB 表：见下文。

### 关联关系

| 维度 | 实现方式 | 基数 |
|---|---|---|
| 主账号 `vendor_account` | `account_id` FK | N:1 |
| 厂商 `vendor` | `vendor_id`（反规范化，等于主账号/板块的 vendor） | N:1 |
| 业务板块 `vendor_product` | `product_id` 引用代码常量 | N:1 |
| 风控隔离组 | `isolation_group_id` FK（可空） | N:1 |
| 服务绑定 `service_binding` | 反向：`pool.credential_services.credential_id` | 1:N |
| 平台业务分类 `platform_category` | **无直接关联**——一个板块可能跨多分类（如 `volc.speech` 同含 asr/tts），分类是 binding 级而非凭据级；从 binding 的 capability 反查 | — |
| 平台 SKU `platform_service` | **无直接关联**——SKU 通过 `(vendor_product_id, capability, model)` 三元组在调度时匹配到 binding，不直接绑凭据 | — |

### 不变量

- `product_id` 必须存在于代码常量 VendorProduct。
- `vendor_id == accounts[account_id].vendor_id == products[product_id].vendor_id`（应用层校验三者一致）。
- 凭据归档后其下所有 bindings 自动停用；凭据**不支持改主账号或改板块**，需新建 + 归档旧的。
