# 五层模型重构实施计划

- 目标：把现有 v0.1（Provider × PoolAccount × ApiKey 三层平铺）重构到 v0.2（Vendor / VendorAccount / VendorProduct / Credential / ServiceBinding 五层）
- 数据结构定义：[`03-核心数据结构设计-v2.md`](03-核心数据结构设计-v2.md)
- UI 参考：[`/preview/redesign-pool.html`](../preview/redesign-pool.html)

## 0. 总体策略

| 项 | 决定 |
|---|---|
| 项目阶段 | 仍在 dev，无线上数据 |
| 迁移方式 | **直接重建**（drop 旧表 + create 新表 + 重新 seed），不双写、不做向后兼容 |
| 提交粒度 | 9 个阶段，每阶段一个独立 PR；每个 PR 自身可编译可测试，可独立回滚 |
| 阻塞关系 | 阶段大致顺序执行；UI 与后端 API 阶段（6 / 7）可并行 |
| 估时 | 单人 10–12 工作日 |
| 风险 | 阶段 4（适配器按板块重组）触及最多代码、最容易出错 |

## 2. 详细阶段

### 阶段 1 — 代码常量层

**目标**：把厂商、业务板块、平台分类、上游能力四个字典固化到 Go 代码里。

**主要改动**：
- 新建 `internal/catalog/category.go` —— `PlatformCategory` map（4 项：llm/asr/tts/mt）
- 新建 `internal/catalog/vendor.go` —— `Vendor` map（6 项：volc/aliyun/tencent/openai/anthropic/deepseek）+ 每个 vendor 的 `MasterAuthSchema`
- 新建 `internal/catalog/product.go` —— `VendorProduct` map（12 项），每个含 `CredentialSchema` + `AllowedCapabilities` + `ProtocolFamily`
- 重写 `internal/catalog/capability.go`（已存在 `internal/capability/capability.go`，并入或导入），13 项 Capability 常量带 `Category` + `BillingUnit`
- 暴露只读 admin API：`GET /api/admin/catalog/{vendors,products,categories,capabilities}`，加在 `internal/admin/catalog.go`
- `configs/providers/*.yaml` 暂时保留作为协议适配的外部配置（base_url、endpoint 模板），但**厂商列表来源切到代码常量**

**验收**：
- 单测：`catalog.Vendors["volc"].Products` 应有 3 项；`catalog.Products["volc.ark"].AllowedCapabilities` 应包含 chat/embedding/vision/image_gen
- 启动时 `loader.go` 校验：每个 product 的 vendor 必须在 Vendors 里、每个 capability 必须在 Capabilities 里
- 调 `/api/admin/catalog/products` 返回 12 项 JSON

### 阶段 2 — DB schema 重建

**目标**：drop 掉 v0.1 的 pool/catalog 全部表，按 v0.2 文档建新表。

**主要改动**：
- 新建 `migrations/00008_pool_v2.sql`：
  ```
  -- +goose Up
  DROP TABLE pool.account_events, pool.api_keys, pool.accounts CASCADE;
  -- 保留 pool.risk_isolation_groups（vendor_id 字段需要 ALTER：从原 provider_id 改名）
  CREATE TABLE pool.vendor_accounts ( ... );
  CREATE TABLE pool.credentials ( ... );
  CREATE TABLE pool.credential_services ( ... );
  CREATE TABLE pool.credential_events ( ... );
  ```
- 新建 `migrations/00009_catalog_v2.sql`：drop 旧 catalog.providers/models/model_mappings/pricing/capabilities，建 `catalog.platform_services` + `catalog.platform_pricing`
- 新建 `migrations/00010_metering_columns.sql`：给 `metering.call_logs` 加 `platform_service_id` / `vendor_account_id` / `credential_id` / `binding_id` 列；旧列（provider_id / pool_account_id）保留兼容期但加注释 `-- DEPRECATED`，下个 release 删除
- 更新 `migrations/00001_init_schemas.sql` 的注释（不改结构）

**验收**：
- `goose up` / `goose down` 双向跑通
- `psql \d+ pool.credentials` 字段、索引、约束全部对得上文档
- 跑现有的 `account_test.go` 等会失败——预期，下阶段修复

### 阶段 3 — Repo + Service 层

**目标**：仓储层按新表重写。Service 暴露的接口尽量贴 admin API 的形状。

**主要改动**：
- `internal/pool/repo/`：
  - 新建 `vendor_account_repo.go` —— `Create/Get/List/UpdateStatus/RefreshBalance/Archive`
  - 新建 `credential_repo.go` —— `Create/Get/List/UpdateHealth/Archive` + by-account / by-product 查询
  - 新建 `binding_repo.go` —— `Create/Get/List/Pick`（调度热路径）/`UpdateUsage/UpdateHealth/Archive`
  - 新建 `event_repo.go` —— 追加型，凭据级 + binding 级
- `internal/pool/service.go`：编排层，处理"创建凭据 = 1 INSERT credentials + N INSERT credential_services 单事务"这种事务边界
- `internal/catalog/repo/`：
  - 新建 `platform_service_repo.go`
  - 新建 `pricing_repo.go`（带 effective_from 历史查询）
- `internal/catalog/service.go`：暴露 `GetCurrentPrice(skuID, at)` / `LookupRoute(skuID) → (productID, capability, model)`
- 删除 `internal/pool/probe.go` 里依赖旧表的部分（probe 逻辑保留，挪到 service 层）

**验收**：
- 单测覆盖：每个 repo 的 CRUD + 边界条件（同名冲突、外键违反、空查询）
- 集成测：service 层"创建凭据 + 4 服务绑定"在事务失败时回滚干净

### 阶段 4 — 适配器按板块重组（最重）

**目标**：把现在按 vendor 平铺的 `internal/provider/{anthropic,deepseek,volc,...}` 重组为按 vendor.product 嵌套的 `internal/upstream/{vendor}/{product}/`。

**主要改动**：
- 新建包结构：
  ```
  internal/upstream/
    iface.go             # Adapter 接口：CredentialSchema, Probe, Endpoint, Invoke
    registry.go          # product_id → Adapter 映射
    volc/
      ark/adapter.go
      speech/adapter.go
      translate/adapter.go
    aliyun/
      dashscope/adapter.go
      nls/adapter.go
      mt/adapter.go
    tencent/
      hunyuan/adapter.go
      speech/adapter.go
      mt/adapter.go
    openai/api/adapter.go
    anthropic/api/adapter.go
    deepseek/api/adapter.go
  ```
- 公共能力放 `internal/upstream/shared/openai_compat/`，被 deepseek.api / aliyun.dashscope 复用
- 旧 `internal/provider/` 删除（包括 `manager.go` / `manager_test.go`）
- 启动时 `registry.Register("volc.ark", &arkAdapter{})` 等
- 每个 adapter 实现统一接口：
  ```go
  type Adapter interface {
      ProductID() string
      CredentialSchema() []FieldSpec
      Probe(ctx, creds) error
      Endpoint(capability) string
      Invoke(ctx, capability, creds, payload) (*Response, error)
  }
  ```

**验收**：
- 12 个 adapter 各自有 unit test（参照现有 `test/golden/`）
- registry 启动时校验：所有代码常量 `VendorProduct.ID` 必须有 adapter 注册，否则 panic
- `test/golden/` 里现有的 volc/anthropic/deepseek 测试用例迁移到新路径并通过

**风险**：
- 不同板块的 endpoint URL 模板差异大（火山 RPC vs OpenAI REST vs WebSocket），iface.go 设计要够通用——可能需要 `Invoke` 之外再加 `StreamInvoke` / `WebsocketInvoke` 形式
- 公共配置（base_url、超时）从 `configs/providers/*.yaml` 读还是写到代码里？建议 yaml 保留为运行时可热更的部分，不放代码常量

### 阶段 5 — 调度器路由

**目标**：网关收到请求后，从 `platform_service_id` 路由到一条 binding。

**主要改动**：
- `internal/gateway/`：入口层解析请求中的 `model`/`platform_service_id` → 查 `catalog.platform_services` → 拿到 (product_id, capability, model)
- `internal/scheduler/picker.go`：把 `Pick(ctx, providerID, capability)` 改成 `Pick(ctx, productID, capability)`，查 `pool.credential_services` 并按 (health DESC, cost_basis ASC, daily_used ASC) 排序
- `internal/scheduler/scorer.go`：评分维度更新（同主账号优先 / 同隔离组冷却扩散）
- 调度命中 binding 后，`Invoke` 时从 vault 取该 binding 对应 credential 的 auth payload，调对应 adapter

**验收**：
- 单测：`scheduler/picker_test.go` 改写覆盖 5 种场景（健康降级、cost 优先、隔离组冷却、quota 耗尽、所有 binding 不可用）
- 集成测：`gateway` 收到 SKU 请求 → 真正命中 binding → 模拟 adapter 返回成功 → metering 写入 platform_service_id + binding_id + cost

## 4. 风险与回滚

| 风险点 | 缓解 |
|---|---|
| 阶段 4 适配器接口设计不通用 | 先用 volc.speech（最特殊，含 WS+签名）和 openai.api（最普适）两端各跑通一个，再拓展其它板块 |
| 阶段 2 migration 0008/0009 一次 drop 太多表 | 在 dev 环境用一次性 reset 脚本（drop schema + goose up），不上 staging 之前不动 |
| 阶段 5 调度器灰度切换 | dev 阶段直接全切；如果未来要上 staging，加 `feature_flag.scheduler_v2` 双跑 7 天比对 |
| 阶段 6/7 接口契约漂移 | 每个 admin 接口写 OpenAPI 描述（或直接用 zod schema），前后端同步 |
| 阶段 4 旧 golden test 覆盖不到新代码 | 阶段开始前先把 `test/golden/_framework` 跑一遍存 baseline，迁移后回放 |

**回滚策略**：每阶段一个 PR，回滚 = `git revert <merge-commit>` + `goose down` 对应 migration。阶段 4 内部如果某个 adapter 写挂了，单独回退该子目录的 commit；其它已迁的不受影响。
