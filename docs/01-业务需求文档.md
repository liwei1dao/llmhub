## 1. 产品定位

### 1.1 一句话描述

LLMHub 是一个**AI 能力聚合与分销平台**，覆盖大模型、语音、翻译、多模态等全栈 AI 能力，为中小开发者和企业提供"零门槛、按量付费"的多厂商统一接入服务。

### 1.2 覆盖的 AI 能力（Capability Matrix）

| 能力域 | 子能力 | 典型上游 | MVP/V1/V2 |
|--------|--------|---------|-----------|
| **LLM 文本生成** | Chat Completions / Tool Use / 结构化输出 | 火山方舟、DeepSeek、阿里百炼、Anthropic | MVP |
| **Embedding** | 文本向量化 / 重排序 | OpenAI兼容各家、BAAI/bge、Qwen-embedding | V1 |
| **ASR 语音识别** | 流式 / 非流式 / 长语音 / 多语种 | 火山 ASR、阿里 NLS、讯飞、Azure、Deepgram | V1 |
| **TTS 语音合成** | 流式 / 音色克隆 / 情感控制 | 火山 TTS、阿里 CosyVoice、ElevenLabs、MiniMax | V1 |
| **实时端到端翻译** | 语音→语音 / 语音→文本 同传 | 火山、讯飞、Azure Speech Translation | V2 |
| **机器翻译（文本）** | 单句 / 文档 / 术语表 | DeepL、Google、百度、有道、火山 | V1 |
| **图像生成** | 文生图 / 图生图 / 图像编辑 | 即梦、通义万相、Flux、DALL·E | V2 |
| **视觉理解（VLM）** | OCR / 图片问答 / 视频理解 | Qwen-VL、豆包 VLM、GPT-4o 类 | V1 |
| **Audio 理解** | 音频分类 / 说话人分离 / 情感识别 | 火山、阿里 | V2 |

### 1.3 价值主张

- **对用户**：一次接入，全栈 AI 能力可用；免签企业合约、免套餐门槛、按调用量计费；统一协议，一行代码切换上游。
- **对平台**：用聚合规模吃批零差价、用账号池化吃利用率红利、用快速接入能力吃新服务的时间窗口，**能力域越多 → 用户黏性越高 → ARPU 越高**。

### 1.3 商业模式

**混合运营**：

| 收入路径 | 占比目标（1 年内） | 说明 |
|----------|------------------|------|
| 企业批发套餐分销 | 60%+ | 与厂商签商务合同，拿企业价批发，零售按量卖 |
| 个人付费账号聚合 | 20-30% | 规避单账号额度/QPS 限制、区分成本档 |
| 新号免费额度利用 | 10%（冷启动期 40%） | 冷启动补血，规模化后比例下降 |
| 合作伙伴转售（后期） | — | 官方 Reseller 渠道 |

**利润模型**：

```
平台毛利 = 用户付费总额 − 上游成本（账号采购 + 调用成本）
         − 运营成本（服务器、风控、对账、客服）
         − 风险准备金（封号、退款、合规）
```

### 1.4 和竞品的差异

| 维度 | OpenRouter | Portkey | 国内中间站 | **LLMHub** |
|------|-----------|---------|-----------|-----------|
| 主打市场 | 海外 | 企业治理 | 灰色短周期 | 国内中小开发者 |
| **能力覆盖** | 仅 LLM 文本 | 仅 LLM 文本 | 多为 LLM | **LLM + 语音 + 翻译 + 多模态全栈** |
| 路由策略 | 成本/延迟 | 可观测+治理 | 单一上游 | **按能力域分层 + 风险分级路由** |
| 账号池构成 | 官方渠道 | 用户自带 | 纯羊毛 | **T1 企业 + T2 付费 + T3 新号混合** |
| 协议 | OpenAI 兼容 | OpenAI 兼容 | 多样 | **OpenAI + Anthropic 双协议 + 能力域统一协议** |
| 合规度 | 高 | 高 | 低 | 中高（主体隔离、分层运营） |

## 2. 用户画像

### 2.1 目标用户

**P0（冷启动期主要目标）**

1. **独立开发者 / 个人创业者**
   - 想做 AI 产品但不想先掏 10 万买套餐
   - 对接入成本敏感，希望换供给不改代码
   - 价格敏感，能接受"便宜 + 偶尔波动"

2. **小型创业公司（5-30 人）**
   - 有业务场景但用量还不稳定
   - 不想签年合约锁定厂商
   - 需要发票，对合规有基本要求

**P1（业务成熟后拓展）**

3. **中型 SaaS 公司**
   - 用量上规模后开始算成本账
   - 需要多厂商路由/降级能力
   - 需要治理、审计、用量看板

4. **教育 / 研究用户**
   - 短期、突发性需求
   - 对模型多样性需求高

### 2.2 用户核心诉求

按优先级：

1. **价格便宜 + 计费透明**
2. **接入简单**（OpenAI SDK 换 base_url）
3. **稳定可用**（单家故障不影响业务）
4. **模型多样**（一个账户用遍火山/阿里/DeepSeek/Claude 等）
5. **账单清晰 + 发票合规**

## 3. 功能范围（MVP → V2）

### 3.1 MVP（0-3 个月）· 文本域全覆盖

**必做（文本）**

- [ ] 用户注册登录（邮箱 + 手机号）
- [ ] 充值 / 账户余额 / 消费明细
- [ ] API Key 管理（用户级 + 能力域粒度授权）
- [ ] Chat Completions 接口（OpenAI 协议兼容）
- [ ] Streaming 支持（SSE）
- [ ] 机器翻译接口（文本 → 文本）——复用 LLM 或专用翻译 API
- [ ] Embedding 接口（如果接入成本低，前置到 MVP）
- [ ] 后台账号池管理（Tier 2+3，**账号按能力域打标**）
- [ ] 基础调度器（按能力域 + 余额 + 健康度打分）
- [ ] 故障自动切换（429/401 自动换号重试）
- [ ] 基础计量（按不同能力域的计价单位：Token / 字符 / 秒 / 次）
- [ ] 接入 2-3 家厂商（火山方舟、DeepSeek、阿里百炼）

**不做（MVP 明确延后）**

- 自研 SDK（一期用各厂商原生 SDK 兼容协议）
- 音频能力（ASR/TTS）
- 实时翻译 / 同传
- 图像 / 视觉理解
- 工单系统（用邮箱客服）

### 3.2 V1（3-6 个月）· 语音域 + 视觉理解

**新增能力**

- [ ] Anthropic Messages 协议兼容
- [ ] **ASR 语音识别**
  - [ ] 非流式（文件上传转写）—— HTTP
  - [ ] 流式（WebSocket）—— 使用上游原生临时 Token 机制
  - [ ] 多语种、长音频分段
- [ ] **TTS 语音合成**
  - [ ] 非流式（一次生成完整音频）
  - [ ] 流式（SSE / WebSocket 边合成边播放）
  - [ ] 音色列表统一抽象（跨厂商音色映射）
- [ ] **视觉理解（VLM）**：图片问答、OCR
- [ ] 工具调用（Function Calling）统一协议
- [ ] 账号池 Tier 1（企业账号）接入
- [ ] 按能力域的独立计费规则（Token / 音频秒数 / 字符 / 次数）
- [ ] 多维用量看板（按能力域 / 模型 / 时间）
- [ ] 发票开具流程
- [ ] Webhook 通知（余额低、调用失败）

### 3.3 V2（6-12 个月）· 多模态 + 实时端到端

**新增能力**

- [ ] **实时端到端翻译**（语音 → 语音 / 语音 → 文本 同传）
  - [ ] 对接厂商原生 RTC / 实时流式接口
  - [ ] 音色保留 / 情感保留
- [ ] **图像生成**（文生图 / 图生图 / 编辑）
- [ ] **音频理解**（说话人分离、情感识别）
- [ ] **视频理解**（V2 后期）
- [ ] 企业版（子账号、预算、审计日志）
- [ ] 合作伙伴分销资质对接
- [ ] 海外版（支持 Stripe、接入 OpenAI/Anthropic/ElevenLabs 官方）
- [ ] 官方 SDK（Python / Node / Go）——封装多能力域统一接口

## 4. 关键业务流程

### 4.1 用户侧：从注册到调用

```
注册 → 邮箱/手机验证 → 实名（可选，影响 QPS 上限）
     → 充值（预付余额）
     → 创建 API Key（指定用途标签）
     → 使用 API Key 调用 /v1/chat/completions
     → 平台鉴权 → 路由到上游 → 流式返回
     → 计量 → 扣费 → 账单更新
```

### 4.2 管理侧：账号池运营

```
【录入】
  运营在后台录入账号 → 选择 Tier / Provider → 填入凭证
  → 系统自动拉取账号余额 → 进入"养号期"（Tier 3）/ 直接上线（T1/T2）

【调度】
  用户请求 → 风险评估 → 选 Tier 偏好 → 从池子挑账号
  → 上游调用 → 记录流水 → 健康度更新

【监控】
  实时监控各账号健康度、余额、QPS、失败率
  → 异常自动熔断 + 告警
  → 每日账单对账（上游账单 vs 平台记录）

【退役】
  余额耗尽 / 到期 / 触发风控 → 账号置为 archived
  → 不再被调度器选中，留存流水数据备审
```

### 4.3 故障切换流程

```
请求 → Provider A 账号 X → 返回 429/5xx
     → 账号 X 健康度下降 → 进入 cooling
     → 立即从同 Provider 的其他账号挑选（同 Tier 优先）
     → 若同 Provider 无可用 → 跨 Provider fallback（如配置允许）
     → 重试上限 N=3，仍失败则返回用户错误
     → 整个流程用户感知到的延迟 < 2s
```

## 6. 合规与风险

### 6.1 必须守住的底线

1. **主体隔离**：羊毛号相关业务使用独立法律主体、域名、支付通道
2. **协议透明**：用户协议声明本平台为聚合服务，不对特定上游 SLA 做承诺
3. **敏感信息**：不留用户 prompt 明文日志（除非用户主动授权用于调试）
4. **账号身份**：不使用伪造实名信息注册上游账号
5. **内容安全**：对调用内容做上游厂商合规要求的基础过滤（敏感词、政治、色情）

### 6.2 业务风险与应对

| 风险 | 概率 | 影响 | 应对 |
|------|------|------|------|
| 上游厂商风控大面积封号 | 中 | 高 | 多 Tier 隔离、跨 Provider 分散、风险准备金 |
| 价格战导致毛利被压缩 | 高 | 中 | 提升服务能力、锁定 T1 长约折扣 |
| 厂商停止新号免费额度 | 高 | 低（若羊毛占比小） | 控制 T3 占比不超总营收 40% |
| 合规监管变化 | 低 | 高 | 主体隔离、合规条款预设 |
| 用户恶意消耗（拉爆账号） | 中 | 中 | 用户侧限流、风险评分、预扣费 |

## 7. MVP 成功标准

**定量指标（3 个月内）**

- 注册用户 ≥ 1000
- 付费用户 ≥ 100
- 月流水 ≥ 10 万元
- 毛利率 ≥ 25%
- 账号池规模：T1 ≥ 2 家签约，T2 + T3 ≥ 50 个账号
- 平台可用性 ≥ 99%

**定性指标**

- 有 10+ 真实用户 case（采访 / 反馈）
- 至少 1 篇社区/媒体自来水传播
- 故障切换机制经过至少 3 次真实故障验证

1. **域名/品牌**：主品牌名、域名尚未定
