# LLMHub 能力域专项设计

- 版本：v0.1
- 日期：2026-04-22

平台不是单一 "LLM 代理"，而是**多能力域的 AI 聚合器**。不同能力域（LLM / ASR / TTS / 翻译 / 同传 / 视觉）在协议、传输、计费、账号池能力匹配上差异巨大，需要在统一抽象层下做针对性设计。

---

## 1. 能力域抽象

### 1.1 Capability Domain 定义

每个能力域是一个独立的"服务类型"，拥有自己的协议、计价单位、SLA 要求。

```yaml
capability_domains:
  - id: chat
    category: llm
    transport: [http, sse]
    billing_unit: token  # (prompt_tokens, completion_tokens)
    streaming: optional

  - id: embedding
    category: llm
    transport: [http]
    billing_unit: token  # 仅 input

  - id: asr
    category: audio
    sub_modes: [batch, streaming]
    transport: [http, websocket]  # batch=HTTP, streaming=WS
    billing_unit: second  # 或 request 按上游定

  - id: tts
    category: audio
    sub_modes: [non_streaming, streaming]
    transport: [http, sse, websocket]
    billing_unit: char  # 通常按字符或合成秒数

  - id: translate_text
    category: translation
    transport: [http]
    billing_unit: char

  - id: translate_realtime
    category: translation
    sub_modes: [speech_to_text, speech_to_speech]
    transport: [websocket, webrtc]
    billing_unit: second

  - id: image_generation
    category: vision
    transport: [http]
    billing_unit: image  # 按张数 × 分辨率

  - id: vlm
    category: vision
    transport: [http, sse]
    billing_unit: token  # 通常上游用 token 计费，vision_tokens 单独

  - id: ocr
    category: vision
    transport: [http]
    billing_unit: page
```

### 1.2 能力域与 Provider 的多对多关系

一个 Provider 通常只覆盖若干能力域。账号池中每个账号需要标注**该账号下哪些能力域可用**。

```
Provider "火山方舟" 能力：chat, embedding, asr, tts, vlm, image_generation
Provider "DeepSeek" 能力：chat, embedding
Provider "ElevenLabs" 能力：tts（独强，音色丰富）
Provider "Deepgram"  能力：asr（多语种领先）
Provider "DeepL"     能力：translate_text（译文质量顶级）
```

**设计原则**：每接入一个新 Provider，标注其**支持的能力域清单 + 每个能力的特长/短板**，调度器按能力+质量偏好分配。

## 2. 能力域专项设计

### 2.1 Chat / Embedding（已在主文档覆盖）

- 协议：OpenAI 兼容为主
- 计费：按 Token
- 流式：SSE
- 详见 [04-模块接口设计.md](04-模块接口设计.md)

### 2.2 ASR 语音识别

#### 2.2.1 两种子模式

| 模式 | 协议 | 场景 |
|------|------|------|
| **batch（非流式）** | HTTP POST，上传音频文件 → 返回文本 | 会议录音转写、音视频字幕 |
| **streaming（流式）** | WebSocket，边推音频边返回识别结果 | 实时语音输入、直播字幕 |

#### 2.2.2 统一接口设计

**Batch 模式**：

```
POST /v1/audio/transcriptions        # 沿用 OpenAI 风格
Content-Type: multipart/form-data

file: (音频文件)
model: whisper-class                 # 逻辑模型
language: zh                         # 可选
response_format: json | verbose_json | srt | vtt
timestamp_granularity: [word, segment]
```

响应：

```json
{
  "text": "...",
  "segments": [...],
  "duration": 32.4,
  "llmhub_meta": {
    "provider": "volc",
    "billed_seconds": 32,
    "request_id": "req_xxx"
  }
}
```

**Streaming 模式**（WebSocket）：

```
WS wss://api.llmhub.io/v1/audio/transcriptions/stream?token=<短时令牌>

客户端 → 服务端（二进制帧）：PCM 或 Opus 音频
服务端 → 客户端（JSON 帧）：
  {"type":"partial","text":"..."}       // 临时识别结果
  {"type":"final","text":"...","start":0.0,"end":2.3}
  {"type":"usage","billed_seconds":30}
  {"type":"done"}
```

#### 2.2.3 关键技术点

- **短时令牌机制**：WebSocket 不走代理模型优先。用户 SDK 先调 HTTP 换 `session_token`（5 分钟 TTL），再连 WebSocket。服务端签 token 时内置"允许的能力域 + 模型 + 预扣费上限"。
- **代理兜底**：若上游 ASR 不支持临时凭证（常见），走薄代理中继（gateway 处理 WS 透传）。
- **切流策略**：ASR 调用中途不切账号（音频上下文会断），宁可拿当前账号重试。
- **计费精度**：按实际音频秒数（精确到 0.1s），非按 wall-clock 时间。

### 2.3 TTS 语音合成

#### 2.3.1 两种子模式

| 模式 | 协议 | 场景 |
|------|------|------|
| **non_streaming** | HTTP POST → 返回整段音频 | 短文本合成、异步生成 |
| **streaming** | SSE 或 WebSocket | 实时播报、AI 对话回复 |

#### 2.3.2 统一接口

```
POST /v1/audio/speech
{
  "model": "tts-class-hd",
  "input": "要合成的文字",
  "voice": "llmhub-voice-001",   # 平台统一音色 ID
  "response_format": "mp3|wav|opus|pcm",
  "speed": 1.0,
  "stream": true,
  "emotion": "neutral"            # 可选：neutral/happy/sad/angry
}
```

#### 2.3.3 音色抽象

音色是 TTS 最头疼的问题：各厂商音色命名/风格完全不同。

**设计策略**：

- 平台维护一张"逻辑音色表" `catalog.voices`
- 每个逻辑音色映射到多家厂商的具体音色（按语言/性别/风格）
- 用户选音色时看到的是平台统一命名（如 `llmhub-female-mandarin-warm`）
- 后端根据调度结果自动映射到上游音色 ID

```
voice_id: llmhub-female-mandarin-warm
  ├─ volc:  BV002_streaming
  ├─ minimax: male-qn-qingse
  └─ aliyun: siyue
```

#### 2.3.4 流式 TTS 关键点

- **首音频 chunk 延迟** 是核心体验指标，目标 < 300ms
- 上游 SSE/WS 流式 chunk 透传到用户
- **中途切换账号基本不可能**（音色要连续），失败就直接报错让用户重试

### 2.4 文本翻译

#### 2.4.1 两条实现路径

| 路径 | 优势 | 劣势 |
|------|------|------|
| **专用翻译 API**（DeepL、Google、火山 MT） | 质量稳、术语表支持、批量快 | 接入多家、计费按字符 |
| **LLM 翻译**（给 Chat 模型发翻译 prompt） | 一个通道多用、风格控制灵活 | 质量波动、成本高、慢 |

**策略**：两种都提供，用户可选 `engine: "mt" | "llm"`，默认 `mt`。

#### 2.4.2 接口

```
POST /v1/translations
{
  "text": "Hello, how are you?",     # 或 texts: [...]
  "source_lang": "auto",
  "target_lang": "zh",
  "engine": "mt",                    # mt | llm
  "glossary_id": "gls_xxx",          # 可选术语表
  "preserve_formatting": true        # HTML/Markdown 标签保留
}
```

响应：

```json
{
  "translations": [
    {"text": "你好，你好吗？", "detected_source": "en"}
  ],
  "llmhub_meta": {
    "engine": "mt",
    "provider": "deepl",
    "billed_chars": 20
  }
}
```

#### 2.4.3 术语表

- 用户可上传自定义术语表（CSV / JSON）
- 平台维护术语表 ID，调用时引用
- 针对专用 MT 翻译：若上游支持术语表 → 透传；若不支持 → 平台在 pre/post 阶段做替换

### 2.5 实时端到端翻译（同传）

这是最复杂的能力域，V2 才做。

#### 2.5.1 模式

| 模式 | 输入 | 输出 | 场景 |
|------|------|------|------|
| speech → text | 音频流 | 翻译文本流 | 实时字幕 |
| speech → speech | 音频流 | 翻译音频流（保持音色） | AI 同传、跨国会议 |

#### 2.5.2 协议

仅 WebSocket / WebRTC，不提供 HTTP 模式。

```
WS wss://api.llmhub.io/v2/translate/realtime?token=<短时令牌>

客户端 → 服务端：
  {"type":"config", "source_lang":"en", "target_lang":"zh",
   "output_mode":"speech", "voice":"llmhub-clone-<id>"}
  <二进制音频帧>

服务端 → 客户端：
  {"type":"partial_text", "lang":"en", "text":"Hello there"}
  {"type":"partial_text", "lang":"zh", "text":"你好"}
  <二进制合成音频帧>
  {"type":"segment_done", "src":"Hello there", "tgt":"你好"}
  {"type":"usage", "billed_seconds":12.3}
```

#### 2.5.3 关键设计

- **流水线组合**：如果上游不提供 end-to-end 同传，平台**自己做流水线**：ASR → MT → TTS，在 gateway 层编排。这是 LLMHub 的独特价值（厂商只能卖单一能力，我们卖组合）。
- **延迟预算**：首译文 < 1.5s，首合成音 < 2.5s。每段不超过端到端 3.5s。
- **音色克隆**：用户可上传 30 秒语音样本，平台调用厂商的音色克隆 API，生成专属音色 ID 供同传使用。
- **账号绑定**：同传会话期间一个账号到底，不切换。

### 2.6 图像生成 / 视觉理解

#### 2.6.1 图像生成

```
POST /v1/images/generations
{
  "model": "image-class-sdxl",
  "prompt": "...",
  "negative_prompt": "...",
  "size": "1024x1024",
  "n": 1,
  "response_format": "url | b64_json",
  "style": "photographic"
}
```

**计费**：按"张数 × 分辨率档位"，不按 Token。
**存储**：生成结果默认存对象存储 1 小时（可用过期 URL 返回），避免大数据走网关重复回传。

#### 2.6.2 视觉理解 / VLM

沿用 OpenAI vision 消息格式（messages 里带 image_url）：

```json
{
  "model": "vlm-class",
  "messages": [
    {"role":"user","content":[
      {"type":"text","text":"这张图里有什么？"},
      {"type":"image_url","image_url":{"url":"https://..."}}
    ]}
  ]
}
```

**计费**：图片按"tile 数"折算成 image_tokens，与文本 token 合并计费。

## 3. 对数据结构的影响（新增/修改）

### 3.1 Catalog 域扩展

```sql
-- 能力域字典
CREATE TABLE catalog.capabilities (
    id              VARCHAR(32) PRIMARY KEY,        -- chat / asr / tts / translate_text / ...
    category        VARCHAR(16) NOT NULL,           -- llm / audio / translation / vision
    display_name    VARCHAR(64) NOT NULL,
    billing_unit    VARCHAR(16) NOT NULL,           -- token / second / char / image / page / call
    sub_modes       TEXT[] DEFAULT '{}',            -- batch / streaming / ...
    transports      TEXT[] DEFAULT '{http}',        -- http / sse / websocket / webrtc
    description     TEXT
);

-- 逻辑音色（跨厂商抽象）
CREATE TABLE catalog.voices (
    id              VARCHAR(64) PRIMARY KEY,
    display_name    VARCHAR(128),
    language        VARCHAR(16),
    gender          VARCHAR(8),
    style_tags      TEXT[] DEFAULT '{}',  -- warm / newsreader / child / energetic
    sample_url      TEXT,
    is_premium      BOOLEAN DEFAULT FALSE,
    status          VARCHAR(16) DEFAULT 'active',
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- 音色 → 厂商具体音色映射
CREATE TABLE catalog.voice_mappings (
    id              BIGSERIAL PRIMARY KEY,
    voice_id        VARCHAR(64) REFERENCES catalog.voices(id),
    provider_id     VARCHAR(32) REFERENCES catalog.providers(id),
    upstream_voice  VARCHAR(128) NOT NULL,
    priority        SMALLINT DEFAULT 10,
    notes           TEXT
);

-- 术语表
CREATE TABLE catalog.glossaries (
    id              VARCHAR(64) PRIMARY KEY,
    user_id         BIGINT REFERENCES iam.users(id),
    name            VARCHAR(128),
    source_lang     VARCHAR(16) NOT NULL,
    target_lang     VARCHAR(16) NOT NULL,
    entries         JSONB NOT NULL,               -- [{"src":"API","tgt":"接口"}]
    created_at      TIMESTAMPTZ DEFAULT NOW()
);

-- catalog.providers 增加字段
ALTER TABLE catalog.providers
  ADD COLUMN supported_capabilities TEXT[] DEFAULT '{chat}';

-- catalog.models 增加字段：模型归属哪个能力域
ALTER TABLE catalog.models
  ADD COLUMN capability_id VARCHAR(32) REFERENCES catalog.capabilities(id);
```

### 3.2 Pool 域扩展（账号能力标签）

```sql
ALTER TABLE pool.accounts
  ADD COLUMN supported_capabilities TEXT[] DEFAULT '{}';
-- 例如：{'chat','embedding','tts'}

-- 每个账号对每个能力域的专属额度（可选，如果上游按能力域分别计额）
CREATE TABLE pool.account_capability_quotas (
    id              BIGSERIAL PRIMARY KEY,
    account_id      BIGINT NOT NULL REFERENCES pool.accounts(id),
    capability_id   VARCHAR(32) NOT NULL,
    quota_total_cents   BIGINT,
    quota_used_cents    BIGINT NOT NULL DEFAULT 0,
    qps_limit       INT,
    updated_at      TIMESTAMPTZ DEFAULT NOW()
);

CREATE UNIQUE INDEX uq_acc_cap ON pool.account_capability_quotas(account_id, capability_id);
```


### 3.4 定价扩展


API Key 的 `scopes` 字段从"模型名"扩展为"能力域 + 模型":

```json

    "tts:llmhub-female-mandarin-warm",
    "asr:*",
    "translate:*"
  ]
}
```

默认 `*:*`，但用户可按能力域精细化授权。

## 5. 账号池调度的能力域感知

### 5.1 调度决策扩展

```go
func Pick(ctx PickCtx) (*Account, error) {
    // 1. 先按能力域过滤
    candidates := pool.Filter(
        provider_supports_capability(ctx.Capability),
        account.Capabilities().Contains(ctx.Capability),
        status == "active",
        capability_quota_enough(ctx.Capability, ctx.EstimatedUnits),
    )

    // 2. 按能力特长打分
    //    例如：asr 优先用 Deepgram-高精度号，tts 优先用 ElevenLabs 高音质号
    scored := scoreByCapabilityStrength(candidates, ctx.Capability, ctx.QualityTier)

    // 3. 会话黏性：同传 / 流式 ASR 强黏性，不切
    if ctx.Capability in ("asr_stream", "tts_stream", "translate_realtime") {
        enforce_strong_stickiness = true
    }

    return scored[0], nil
}
```

### 5.2 账号入池时的能力验证

运营录入账号时，**自动调用每个能力域的最小测试调用**，探测该账号实际可用的能力域：

```
POST /api/admin/pool/accounts/{id}/probe-capabilities

后端：
  for each declared capability:
    发一个最小成本的测试请求（比如 ASR 传 1 秒空音频）
    记录：可用 / 不可用 / 需配额申请
    结果写入 account.supported_capabilities
```

这避免了"账号说有 TTS 能力，实际上游要单独申请"这种坑。

## 6. 计费标准化

**核心挑战**：不同能力域不同单位，如何对用户统一展示？

**解决方案**：用户侧账单统一用**"等值 token"**或**"¥"**两种展示方式。

- 账单详情：按调用显示真实单位（"此次 ASR：32.4 秒，¥0.12"）
- 账单总览：统一用金额
- 仪表盘图表：可切换"调用数 / 费用 / 各能力域消耗占比"

预扣费策略因能力域而异：

| 能力域 | 预扣估算 |
|--------|---------|
| chat | max_tokens × 价格 × 2（安全系数） |
| asr batch | 音频时长秒数 × 价格 |
| asr stream | `initial_freeze = 300s × 价格`，低于阈值补扣 |
| tts | 输入文本字符数 × 价格 |
| translate | 源文本字符数 × 价格 |
| image gen | n × 分辨率单价 |
| realtime translate | `initial_freeze = 600s × 价格`，滚动补扣 |

## 7. 对预览 UI 的影响

用户控制台需要按**能力域分区展示**：

- 顶部 tab：Chat / Speech / Translation / Vision / Image
- 各能力域独立：模型选择器、调用明细、用量图表
- 全局：总余额、API Key（可跨能力域或单能力域授权）

运营后台账号池表格新增：

- **"支持能力"**列（多标签展示）
- **"能力维度额度"**展开（点击账号展开看各能力域剩余额度）
- **按能力域筛选**（只看支持 TTS 的账号）

## 8. 实施优先级建议

**MVP 阶段不要贪多**。能力域扩展按业务价值 + 接入复杂度排序：

| 优先级 | 能力域 | 理由 |
|--------|--------|------|
| MVP | chat, translate_text | chat 是基座，翻译可复用 chat 快速上 |
| V1 早 | embedding, vlm | 同属 LLM 协议栈，成本低 |
| V1 中 | asr (batch), tts (non-stream) | 用 HTTP 起步，不碰 WS |
| V1 晚 | asr (stream), tts (stream) | 上 WS，要解决代理/短时令牌 |
| V2 | image generation, ocr | 多模态生态成熟后 |
| V2 后 | realtime translation | 最复杂，作为差异化杀手锏 |
