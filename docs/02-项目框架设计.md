- 版本：v0.1
- 日期：2026-04-22

## 1. 技术选型

### 1.1 选型原则

- **追求研发速度 > 追求性能极致**（MVP 阶段）
- **成熟技术优先**，避免踩坑拖慢进度
- **可水平扩展**，数据库和网关都要能横向扩容
- **运维简单**，优先用 K8s + 托管组件

### 1.2 技术栈

| 层 | 选型 | 备选 | 理由 |
|----|------|------|------|
| **后端语言** | Go 1.22+ | Rust, Node | 并发强、IO 密集场景性能好、团队熟悉 |
| **Web 框架** | Gin / Fiber | Echo | 生态成熟，MVP 够用 |
| **代理网关核心** | 自研（基于 fasthttp） | Envoy, Nginx Lua | 需要深度定制调度逻辑 |
| **主数据库** | PostgreSQL 16 | MySQL | JSONB、时序扩展、事务能力 |
| **缓存 + 消息** | Redis 7 | — | 调度器热数据、限流、会话黏性 |
| **时序数据** | TimescaleDB (PG 扩展) | ClickHouse | 调用流水用 PG 扩展省运维成本 |
| **消息队列** | NATS JetStream | Kafka | 轻量、够用、运维成本低 |
| **对象存储** | S3 兼容（MinIO 自建 / 阿里云 OSS） | — | 日志归档、发票附件 |
| **配置/密钥** | HashiCorp Vault | AWS Secrets Manager | 上游账号密钥安全托管 |
| **前端框架** | Next.js 15 (App Router) | Remix | SSR + 管理后台够用 |
| **UI 组件** | shadcn/ui + Tailwind | Antd | 定制能力强、审美现代 |
| **监控** | Prometheus + Grafana + Loki | — | 标配 |
| **部署** | K8s (阿里云 ACK) | Docker Compose | 起步单节点 → 平滑扩容 |

### 1.3 关键中间件说明

- **Vault**：上游账号的 API Key / AppKey / Secret 必须加密存放，**数据库里只存 Vault 引用**，业务代码通过 Vault SDK 在运行时拉取。即便 DB 被拖库，密钥也不会泄漏。
- **Redis**：承担账号池调度器的热路径——健康度快照、可用账号列表、会话黏性绑定、QPS 限流滑窗。所有 O(ms) 读操作走 Redis，PG 作持久化。
- **NATS**：调用流水异步写、账单对账事件、账号状态变更广播。解耦网关和下游分析链路。

### 2.2 核心服务拆分

按**限界上下文（DDD）**划分，每个服务独立可部署：

| 服务 | 职责 | 对外接口 |
|------|------|----------|
| **api-gateway** | 用户调用入口、鉴权、限流、协议转换、上游代理、流式转发 | `/v1/*` OpenAI + `/anthropic/*` 协议 |
| **scheduler** | 账号池调度、健康度管理、故障切换决策 | 内部 gRPC |
| **account** | 用户账号、API Key、余额、充值、发票 | `/api/user/*`、`/api/admin/users/*` |
| **pool** | 上游账号池 CRUD、生命周期、风控规则 | `/api/admin/pool/*` |
| **billing** | 预扣费、结算、用量聚合、对账 | 内部事件消费 + `/api/user/usage`、`/api/user/bills` |
| **provider-adapter** | 各厂商协议适配（进程内库，非独立服务） | Go 接口 |
| **admin-web** | 运营后台 UI | Web |
| **user-console** | 用户控制台 UI | Web |

**为什么不一体化？** 调用流量（api-gateway）和管理流量（account/admin）的扩缩容节奏完全不同。网关要能弹性扩到几十个实例，管理后台常年 2 个实例够用。

### 2.3 数据流：一次用户调用的生命周期

```
1. 用户 SDK 发起 POST /v1/chat/completions，Bearer {用户 API Key}
        │
2. api-gateway 鉴权：
   - 查 Redis 缓存的用户 API Key → 用户 ID + 限流状态
   - 命中限流 → 429 返回
        │
3. 计费预检：
   - 请求预估 Token（按 prompt 长度）→ 估价
   - billing.check_and_freeze(user_id, estimated_cost)
   - 余额不足 → 402 返回
        │
4. 路由决策：
   - scheduler.pick(model=gpt-4o-class, risk=medium)
   - 返回 {provider: volc, account_id: xxx, endpoint_id: ep-yyy}
   - Redis 会话黏性：如果用户已有进行中的会话，优先复用同账号
        │
5. 协议转换 + 转发：
   - provider-adapter[volc].translate_request(openai_req) → volc_req
   - 从 Vault 取账号凭证
   - 向上游发起 HTTP（含 SSE）
        │
6. 流式回传：
   - 上游 SSE → gateway 转换为 OpenAI SSE 格式 → 用户
   - 边转发边累计实际 Token
        │
7. 完成/失败处理：
   - 成功 → NATS 发布 "call.completed" 事件
     billing 消费 → 结算实际 cost，释放冻结余额差
     scheduler 消费 → 更新账号健康度 +1
   - 失败（429/5xx）→
     - scheduler 标记账号 cooling
     - 从同 Tier 内挑备用账号重试（上限 3 次）
     - 仍失败 → 返回用户错误 + 退还冻结
        │
8. 流水入库：
   - TimescaleDB 写入 call_log（异步批量）
   - PG 更新用户余额 + 账号使用量（事务）
```

## 3. 模块设计

### 3.1 API Gateway 模块

**职责层次**（从入口到出口）：

```
┌──────────────────────────────────────────┐
│ L1  协议入口（HTTP/SSE/WS）              │
├──────────────────────────────────────────┤
│ L2  鉴权 + 限流（Redis）                 │
├──────────────────────────────────────────┤
│ L3  计费预检（调 billing）               │
├──────────────────────────────────────────┤
│ L4  路由决策（调 scheduler）             │
├──────────────────────────────────────────┤
│ L5  协议转换（provider-adapter）         │
├──────────────────────────────────────────┤
│ L6  上游转发 + 流式处理                  │
├──────────────────────────────────────────┤
│ L7  计量 + 事件发布（NATS）              │
└──────────────────────────────────────────┘
```

**关键技术点**：

- **零拷贝流式转发**：SSE 流直接管道化，尽量不解析每个 chunk 的内容，仅在遇到 `data: [DONE]` 时截取用量信息。
- **连接池复用**：对上游用 keep-alive + HTTP/2，减少握手。
- **Graceful shutdown**：收到 SIGTERM 后停止接新请求，等现有请求（含长流）完成再退出，配合 Fluid Compute / K8s preStop。

### 3.2 Scheduler 模块（账号池调度器）

**核心状态**全部放 Redis，PG 只做持久化：

```
Redis 数据结构：

pool:providers:active                # Set，所有可用 provider
pool:provider:{pid}:accounts         # ZSet，账号ID → 健康度分数
pool:account:{aid}:state             # Hash，账号当前状态快照
pool:account:{aid}:qps:{minute}      # String，分钟级 QPS 计数
session:sticky:{user_id}:{provider}  # String，会话黏性绑定的账号ID
```

**调度决策核心逻辑（伪代码）**：

```go
func Pick(ctx PickCtx) (*Account, error) {
    // 1. 会话黏性优先
    if sticky := redis.Get(fmt.Sprintf("session:sticky:%s:%s", ctx.UserID, ctx.Provider)); sticky != nil {
        if acc := GetAccount(sticky); acc.IsHealthy() {
            return acc, nil
        }
    }

    // 2. 候选集：同 Provider、支持该 model、状态 active、配额足够
    candidates := filterCandidates(ctx.Provider, ctx.Model, ctx.EstimatedTokens)
    if len(candidates) == 0 {
        return nil, ErrNoAccountAvailable
    }

    // 3. 打分：
    //    score = 0.4*health + 0.3*tier_match + 0.2*quota_urgency + 0.1*load_balance
    //    tier_match: ctx.RiskLevel 越低越偏好高 Tier
    //    quota_urgency: 剩余余额越接近阈值分数越高（临期优先消耗，只对 T3）
    //    load_balance: 最近 QPS 越低得分越高（分散负载）
    scored := scoreAll(candidates, ctx)
    best := topK(scored, 1)[0]

    // 4. 绑定会话黏性（TTL = 会话超时）
    redis.SetEX(fmt.Sprintf("session:sticky:%s:%s", ctx.UserID, ctx.Provider), best.ID, sessionTTL)

    return best, nil
}
```

**健康度维护**：

- 调用成功 → 健康度 +1（封顶 100）
- 5xx/超时 → 健康度 -5
- 429 → 健康度 -10，进入 cooling 60s
- 401/403 → 健康度 = 0，账号状态置为 suspected
- 连续 3 次同类失败 → 告警 + 状态 quarantine

### 3.3 Account 模块（用户账号）

- 用户注册/登录（短信验证码 + 邮箱密码）
- 实名认证（V1 加入）
- API Key 管理（支持多 Key、标签、细粒度权限、IP 白名单）
- 充值（支付宝/微信 Native，后期 Stripe）
- 余额变动流水（每笔都有事务性入库）

**安全要点**：

- API Key 存储：`prefix_visible + hash`。显示时 `sk-llmh-1a2b****f8e9`。
- 敏感操作（充值、改密、删 Key）二次验证
- 登录设备管理 + 异地登录风险邮件

### 3.4 Pool 模块（上游账号池）

- 账号 CRUD（含凭证录入、加密入 Vault）
- Tier 分层管理
- 账号生命周期状态机（active / cooling / rate_limited / depleted / quarantine / archived）
- 风控隔离组（risk_isolation_group）
- 余额同步任务（每小时拉上游 API，T1 每 10 分钟，T3 每日）
- 养号策略（Tier 3 新号首周调用行为模拟）

### 3.5 Billing 模块

- 余额冻结/释放（乐观锁 + 行级锁）
- 预估计费（基于 prompt 长度 × 估算倍数）
- 实际计费（基于上游返回的 usage）
- 按日聚合用量（TimescaleDB continuous aggregate）
- 发票开具（V1）
- 对账（每日定时任务，比对上游账单 API）

### 3.6 Provider-Adapter（库，非服务）

每个 Provider 一个 Go package，实现统一接口：

```go
type Provider interface {
    ID() string
    SupportedModels() []Model
    TranslateRequest(openaiReq *OpenAIChatRequest) (*http.Request, error)
    TranslateResponse(resp *http.Response) (*OpenAIChatResponse, error)
    TranslateStreamChunk(chunk []byte) ([]byte, error)  // 流式 chunk 协议转换
    ParseUsage(resp interface{}) (*Usage, error)
    MapError(err error, resp *http.Response) *UnifiedError
}
```

目录结构：

```
internal/adapter/
├── adapter.go              # 统一接口定义
├── registry.go             # 按 provider_id 注册和查找
├── openai_compat/          # 通用 OpenAI 兼容实现（80% 厂商可复用）
├── volc_ark/
├── deepseek/
├── dashscope/              # 阿里百炼
├── anthropic/
└── testdata/               # 协议一致性测试固定数据
```

**YAML 配置驱动**：对完全兼容 OpenAI 协议的厂商，仅需配置文件即可上线，零 Go 代码。

## 5. 安全设计

### 5.1 凭证管理三层隔离

```
第一层：用户侧 API Key
  - 用户自己的调用凭证
  - 存储：prefix + hash，明文仅在创建时一次性返回
  - 权限：绑定用户，限额、IP 白名单、模型白名单

第二层：内部服务间凭证
  - gateway ↔ scheduler ↔ billing 之间用 mTLS
  - 不使用静态 token

第三层：上游厂商凭证
  - AK/SK、API Key 全部存 Vault
  - 数据库只保存 Vault 引用（vault://secret/pool/{account_id}）
  - 运行时按需拉取，短期内存缓存
```

### 5.2 数据安全

- 用户密码：Argon2id
- 传输：全链路 TLS 1.3
- 数据库连接：SSL + CA 校验
- 敏感业务操作：审计日志（不可删除，追加型存储）

### 5.3 风控

- 用户侧：异常登录、异常调用模式（QPS 突增、模型突变、地域突变）
- 账号池侧：上游返回的风控信号（特定错误码 + 关键词）实时处理
- 资金侧：大额充值、异常退款流程触发人工审核
