# LLMHub Go 框架设计与厂商扩展

- 版本：v0.1
- 日期：2026-04-22
- 受众：后端与前端开发工程师
- 目标：定义 Go 单仓结构、厂商/能力域扩展机制、编码规范，让"接入一个新厂商 + 一个新能力域"的工作量可控且不污染核心代码

---

## 1. 设计目标与约束

### 1.1 设计目标

1. **新厂商接入 ≤ 1 人天**：理想情况下只加 1 个目录 + 1 个 YAML，不改核心代码
2. **新能力域接入 ≤ 1 人周**：能力域属于平台级扩展，需要新增协议/路径/计费单元，但不动账号池和调度器
3. **模块边界清晰**：能力域抽象层、厂商适配层、调度层、计费层、账户层相互独立
4. **统一的错误、日志、指标规范**：任何新代码都遵循同一套上下文传递和观测规范
5. **测试友好**：每个厂商都必须有协议一致性测试，每个能力域都有冒烟集

### 1.2 关键约束（已决策 2026-04-22）

- 语言：**Go 1.22+**（**允许并鼓励使用 generics**，尤其在 registry、estimator、结果包装等核心抽象处）
- 仓库：**单仓单 module**，不用 Workspaces
- **服务形态：从第一天就是多服务架构**。`gateway / scheduler / account / billing` 四个核心服务独立二进制、独立镜像、独立部署。服务间通过 gRPC（同步）+ NATS（异步）通信。
- **数据库：PostgreSQL 16 + TimescaleDB 扩展**
  - 自建部署（K8s + CloudNativePG，或云厂商托管 PG）
  - **备选：Supabase**。Supabase 支持 TimescaleDB 扩展，可作为 MVP 提速方案，但我们**只用它的 Postgres**，不用其 Auth / Realtime / Storage，避免厂商锁定。
- **前端：Next.js 15 App Router（SSR/SSG，面向 SEO 优化）** + shadcn/ui + TypeScript
  - 官网/文档/博客独立 SSG 站点（`web/www`），最大化 SEO
  - 用户控制台（`web/console`）、运营后台（`web/admin`）登录后使用，对 SEO 无需求
- **厂商适配按能力域拆子包**：`provider/volc/chat`、`provider/volc/asr` 独立目录
- **CI 强制 golden 测试**：协议一致性测试失败禁止合并

**关键原则**：

1. `internal/` 里所有包都**可以互相调用**，但按层次自上而下依赖：
   `gateway/admin/userapi` → `capability` → `provider` + `pool` + `billing` + `iam` + `wallet` + `catalog` + `metering` → `domain` + `platform`
2. `provider/xxx` 不得直接调用 `scheduler`、`billing`，只处理 HTTP 细节
3. `capability/xxx` 只调 `provider` 接口，不关心具体厂商
4. `platform/` 永远不依赖 `domain/` 之外任何业务代码

## 3. 核心抽象（Go 接口）

### 3.1 能力域抽象 `capability.Capability[Req, Resp]`（泛型）

核心抽象使用 Go 泛型，避免 `any` 到处传导致丢失类型安全。

```go
// internal/capability/capability.go
package capability

import (
    "context"
    "github.com/llmhub/internal/domain"
)

// Capability[Req, Resp] 表示一个 AI 能力域。
// 每个能力域实现自己的请求解析、计费估算、响应构造，类型参数保证调用方拿到强类型的 Req/Resp。
type Capability[Req any, Resp any] interface {
    ID() string
    BillingUnit() domain.BillingUnit
    Estimate(ctx context.Context, req Req) (domain.BillingEstimate, error)
    RequiredTransport() []domain.Transport
}

// Result[T] 统一结果包装
type Result[T any] struct {
    Value    T
    Usage    domain.Usage
    Provider string
    AccountID int64
    Meta     map[string]any
}

// Pipeline[In, Out] 能力组合（例：同传 = ASR.Out → MT.In, MT.Out → TTS.In）
type Pipeline[In any, Out any] interface {
    Execute(ctx context.Context, in In) (Result[Out], error)
}
```

计费估算器也用泛型统一：

```go
// internal/billing/estimator/estimator.go
type Estimator[Req any] interface {
    Estimate(req Req, pricing domain.Pricing) domain.BillingEstimate
}

// 具体实现
type TokenEstimator struct{}
func (TokenEstimator) Estimate(req *domain.ChatRequest, p domain.Pricing) domain.BillingEstimate { ... }

type SecondEstimator struct{}
func (SecondEstimator) Estimate(req *domain.ASRRequest, p domain.Pricing) domain.BillingEstimate { ... }
```

### 3.2 厂商适配抽象 `provider.Provider`

```go
// internal/provider/provider.go
package provider

import (
    "context"
    "io"
    "net/http"

    "github.com/llmhub/internal/domain"
)

// Provider 代表一家上游厂商（火山 / DeepSeek / ElevenLabs / ...）
// 一家厂商可能支持多个能力域，通过各 Capability* 接口的可选实现来声明支持。
type Provider interface {
    ID() string
    SupportedCapabilities() []string
    Config() Config

    // 获取某能力的 handler；返回 nil 表示不支持该能力
    ChatCapability() ChatProvider
    EmbeddingCapability() EmbeddingProvider
    ASRCapability() ASRProvider
    TTSCapability() TTSProvider
    TranslateCapability() TranslateProvider
    ImageCapability() ImageProvider
    VLMCapability() VLMProvider
    RealtimeTranslateCapability() RealtimeTranslateProvider
}

// ChatProvider 是某厂商对 chat 能力域的实现
type ChatProvider interface {
    // 将平台统一请求翻译为厂商请求
    TranslateRequest(ctx context.Context, req *domain.ChatRequest, cred domain.Credential) (*http.Request, error)
    // 解析厂商非流式响应
    TranslateResponse(ctx context.Context, resp *http.Response) (*domain.ChatResponse, error)
    // 流式 chunk 转换
    TranslateStreamChunk(chunk []byte) ([]byte, error)
    // 从响应里解析实际 usage
    ParseUsage(body []byte) (*domain.Usage, error)
    // 错误码映射
    MapError(statusCode int, body []byte) *domain.UnifiedError
}

type ASRProvider interface {
    // batch
    TranscribeBatch(ctx context.Context, audio io.Reader, req *domain.ASRRequest, cred domain.Credential) (*domain.ASRResponse, error)
    // streaming: provider 返回的 WS 连接由 gateway 接管转发
    StreamingEndpoint(ctx context.Context, req *domain.ASRStreamRequest, cred domain.Credential) (*domain.UpstreamStreamEndpoint, error)
    MapError(statusCode int, body []byte) *domain.UnifiedError
}

// 其他 XxxProvider 同理
```

**关键设计**：每个能力域有自己的 Go interface，Provider 实现哪些 interface 就代表支持哪些能力。**不支持的能力返回 `nil`**，路由层检测到 nil 就跳过。

### 3.3 厂商注册表（泛型 Registry）

```go
// internal/provider/registry.go
package provider

import (
    "fmt"
    "sync"
)

// Registry[T] 是一个通用注册表。T 可以是 Provider、Capability 实现等。
type Registry[T any] struct {
    mu    sync.RWMutex
    items map[string]T
}

func NewRegistry[T any]() *Registry[T] {
    return &Registry[T]{items: make(map[string]T)}
}

func (r *Registry[T]) Register(id string, v T) {
    r.mu.Lock()
    defer r.mu.Unlock()
    if _, ok := r.items[id]; ok {
        panic(fmt.Sprintf("%s already registered", id))
    }
    r.items[id] = v
}

func (r *Registry[T]) Get(id string) (T, bool) {
    r.mu.RLock()
    defer r.mu.RUnlock()
    v, ok := r.items[id]
    return v, ok
}

func (r *Registry[T]) All() map[string]T {
    r.mu.RLock()
    defer r.mu.RUnlock()
    out := make(map[string]T, len(r.items))
    for k, v := range r.items {
        out[k] = v
    }
    return out
}

// Provider 专用
var Providers = NewRegistry[Factory]()

type Factory func(cfg Config) (Provider, error)
```

每个厂商主适配器 `adapter.go` 在 `init()` 里注册自己，并从子包聚合各能力实现：

```go
// internal/provider/volc/adapter.go
package volc

import (
    "github.com/llmhub/internal/provider"
    "github.com/llmhub/internal/provider/volc/asr"
    "github.com/llmhub/internal/provider/volc/chat"
    "github.com/llmhub/internal/provider/volc/tts"
)

func init() {
    provider.Providers.Register("volc", func(cfg provider.Config) (provider.Provider, error) {
        return newAdapter(cfg)
    })
}

type adapter struct {
    cfg  provider.Config
    chat *chat.Adapter
    asr  *asr.Adapter
    tts  *tts.Adapter
}

func (a *adapter) ID() string                    { return "volc" }
func (a *adapter) ChatCapability() provider.ChatProvider { return a.chat }
func (a *adapter) ASRCapability()  provider.ASRProvider  { return a.asr }
func (a *adapter) TTSCapability()  provider.TTSProvider  { return a.tts }
// 不支持的能力域返回 nil，路由层会跳过
func (a *adapter) ImageCapability() provider.ImageProvider { return nil }
```

### 3.6 多服务间通信

既然从 Day 1 多服务部署，服务间通信必须定义清晰：

| 通信方向 | 协议 | 说明 |
|---------|------|------|
| gateway → scheduler | gRPC (sync) | `Pick` / `Report` |
| gateway → billing | gRPC (sync) | `Freeze` / `Settle` / `Release` |
| gateway → account | gRPC (sync) | API Key 鉴权（带 Redis 缓存前置） |
| worker → billing | gRPC | 对账任务调用 |
| 各服务 → NATS | pub/sub | 异步事件（call.completed / pool.state_changed / ...） |
| 各服务 → Vault | HTTP | 取凭证（短期内存缓存） |

gRPC 契约全部放 `proto/`，每个服务都用 `buf` 生成自己语言的 stub：

```
proto/
├── buf.yaml
├── buf.gen.yaml
├── scheduler/v1/scheduler.proto
├── billing/v1/billing.proto
└── account/v1/account.proto
```

服务间通信**必须 mTLS**。Vault 签发服务证书，轮换周期 24 小时。

## 6. 数据访问层

### 6.0 数据访问规范

**技术选型**：
- PostgreSQL 16 + TimescaleDB 扩展
- 数据访问使用 **sqlc**（类型安全 + 手写 SQL）
- 连接池 **pgx/v5**
- 迁移工具 **goose**

**备选托管方案**：
- Supabase（支持 TimescaleDB 扩展，免运维，但有厂商锁定）
- 阿里云 PolarDB-PG / RDS PostgreSQL（国内用户低延迟）
- AWS RDS PostgreSQL / Google Cloud SQL（海外）

**约束**：
- 只使用 Supabase 的 Postgres 服务，不使用其 Auth / Realtime / Storage（避免锁定）
- 所有 SQL 放 `internal/{domain}/repo/queries/*.sql`，sqlc 生成对应 Go 代码到 `internal/{domain}/repo/gen/`
- 禁止在 handler / service 层出现 SQL 字符串或 ORM 表达式
- 事务通过 `repo.Tx(ctx, func(tx Queries) error { ... })` 接口传递，service 层不接触 `*sql.Tx` 实例

### 6.1 目录结构示例

```
internal/pool/
├── repo/
│   ├── queries/           # 手写 SQL（sqlc 输入）
│   │   ├── accounts.sql
│   │   ├── api_keys.sql
│   │   └── events.sql
│   ├── gen/               # sqlc 生成（不手改）
│   │   ├── queries.sql.go
│   │   ├── models.go
│   │   └── db.go
│   ├── repo.go            # 封装 Queries，提供领域层友好方法
│   └── tx.go              # 事务辅助
├── service.go
└── ...
```

### 6.2 配置加载与动态更新

#### 三层配置

```
优先级低 → 高
┌─────────────────────────┐
│ 1. 代码默认值             │
├─────────────────────────┤
│ 2. configs/app.yaml     │  (基础配置)
├─────────────────────────┤
│ 3. configs/app.{env}.yaml │ (环境覆盖)
├─────────────────────────┤
│ 4. 环境变量 LLMHUB_*      │ (容器化覆盖)
├─────────────────────────┤
│ 5. DB 中的动态配置表      │ (运行时可改)
└─────────────────────────┘
```

- **静态配置**（端口、连接串、功能开关）走 YAML + ENV
- **动态配置**（Provider 状态、模型映射、定价）走数据库，启动时加载 + 监听 pg NOTIFY 热更新
- **敏感配置**（Vault Token、DB 密码）走容器平台 Secret，不进 Git

### 6.3 Provider 配置加载流程

```
启动时：
  1. catalog.loader 从 configs/providers/*.yaml 读取静态声明
  2. 对比 DB 中 catalog.providers 表，不存在的 INSERT，存在的忽略
  3. provider.Registry 里的 Factory 按 DB 中 status=active 的 provider 实例化
  4. 监听 pg NOTIFY "provider.reload"
       → 收到事件 → 重新 Build provider 实例 → 原子替换指针
```

**热更新的作用**：后台运营修改了某个 Provider 的定价 / 错误映射 / 开关状态，不需要重启服务，秒级生效。

## 7. 一次 Chat 调用的内部流程（代码视角）

```
HTTP POST /v1/chat/completions
     │
     ▼
internal/gateway/openai/router.go  ──>  chat.Handler(w, r)
     │
     ▼
internal/capability/chat/handler.go
     │
     │  1. 解析请求为 domain.ChatRequest
     │  2. middleware 已完成鉴权，context 里有 user_id / api_key_id
     │  3. chat.billing.Estimate(req) → estimate
     │  4. billing.Service.Freeze(user, estimate) → hold_id
     │  5. scheduler.Service.Pick(PickCtx{
     │        Capability: "chat", Model: req.Model, UserID, RiskLevel,
     │     }) → pickResult{ProviderID, AccountID, VaultRef, UpstreamModel}
     │  6. vault.Resolve(vaultRef) → Credential
     │  7. provider.Registry[pickResult.ProviderID].ChatCapability() → chatProvider
     │  8. upstreamReq, _ := chatProvider.TranslateRequest(ctx, req, cred)
     │  9. httpclient.Do(upstreamReq) → upstreamResp
     │ 10. 若 stream=true: 边转发 chunk 边累计 token，结束后 ParseUsage
     │     若非流式: TranslateResponse → 构造 OpenAI 响应 + ParseUsage
     │ 11. 失败且可降级 → scheduler 反馈 + 换号重试（最多 3 次）
     │ 12. 成功 → billing.Settle(hold_id, actualUsage)
     │     失败 → billing.Release(hold_id)
     │ 13. metering.Logger 异步写 call_logs
     │ 14. NATS 发 llmhub.call.completed.v1
     │
     ▼
Response to client
```

每一步都在**独立的包**里完成，通过接口而非具体类型耦合，测试时可以 mock 任意一步。

## 11. 编码规范要点

1. **包名短小清晰**：`pool` / `scheduler` / `billing`，不要 `poolmanager`、`schedservice`
2. **所有公共函数必须接受 `context.Context` 作为第一个参数**
3. **错误包装**：使用 `fmt.Errorf("xxx: %w", err)`，错误类型断言统一走 `errors.As`
4. **禁止全局变量**（除了 registry / metrics / logger 等基础设施）
5. **依赖注入**：通过构造函数 `NewXxxService(deps...)`，不用 container 框架
6. **goroutine 必须 context 化**：没有 cancel channel 或 ctx 的 goroutine 禁止合并
7. **资源释放**：defer close / cancel 靠近资源创建
8. **测试文件**：`_test.go` 必须使用 `t.Parallel()` 除非有顺序依赖
9. **禁止接收者污染**：不要在 struct 方法里改全局状态，除非是注册类
10. **SQL 放在 repo 层**：handler/service 层禁止出现 SQL 字符串

### 13.2 `web/console` —— 用户控制台（登录后）

- App Router + CSR 为主，SEO 无要求
- `middleware.ts` 做鉴权重定向
- 使用 `@tanstack/react-query` + SWR 做数据缓存
- shadcn/ui 组件库

### 13.3 `web/admin` —— 运营后台（内部）

- 限制 IP 白名单访问
- 多角色权限（超级管理员、运营、财务、客服）
- 审计日志集成（关键操作留痕）

### 13.4 三站共享

- **包管理**：pnpm workspace（`web/pnpm-workspace.yaml`），共享 `@llmhub/ui`（组件）、`@llmhub/config`、`@llmhub/types`（从 OpenAPI 生成）
- **类型生成**：后端发布 `openapi.json`，前端 CI 自动跑 `openapi-typescript` 生成 `@llmhub/types`
- **部署**：`www` 独立部署（可用 Vercel / Cloudflare Pages / 自建 Node）、`console` 和 `admin` 可合并也可独立
- **监控**：PostHog（产品分析）+ Sentry（错误监控）

## 15. 决策记录（2026-04-22 已确认）

| # | 决策 | 说明 |
|---|------|------|
| 1 | **多服务从 Day 1** | `gateway / scheduler / account / billing / worker / cli` 各自独立二进制、独立镜像、独立扩缩容；服务间 gRPC + NATS，全程 mTLS |

| 3 | **前端 SEO 优先** | `web/www` 独立 SSG 站点，面向搜索引擎；`console/admin` CSR 登录后使用；Lighthouse ≥ 95 纳入 CI |
| 4 | **数据库 Postgres / Supabase** | 优先自建 PG 16 + TimescaleDB；MVP 阶段可用 Supabase 托管 PG 加速，**只使用其 PG 能力**，不用 Auth/Realtime/Storage |
| 5 | **厂商按能力域拆子包** | `provider/volc/chat`、`provider/volc/asr` 分别独立目录，共享逻辑放 `provider/volc/auth` `provider/volc/errors` |

## 16. 仍需决策的细节问题

以下是上面决策延伸出来的细节，后续实施时再定：

1. **Supabase vs 自建 PG 的最终选择**：Supabase 国内访问延迟？TimescaleDB 具体版本是否满足我们的 continuous aggregate？**建议**：冷启动用 Supabase 快速起步，业务量起来后迁到自建/阿里云 PolarDB-PG。
2. **gRPC 服务发现**：K8s 内置 Service DNS 够用 or 用 Consul？**建议**：K8s DNS。
3. **前端 UI 组件库**：shadcn/ui（copy-paste）vs Radix + 自研？**建议**：shadcn/ui。
4. **文档站技术**：MDX + Next.js 内嵌 vs Mintlify/Nextra？**建议**：内嵌以便与 SEO 站点一体化。
5. **多语言覆盖**：MVP 仅中文，还是中英文都做？**建议**：MVP 中文主 + 英文站架构预留但不翻译。
6. **CI 平台**：GitHub Actions vs GitLab CI？**建议**：GitHub Actions（生态好、与国内镜像仓库打通成熟）。
